# System Architecture

Parts Catalog implements a modern three-tier architecture separating the presentation layer (Vue.js), business logic layer (Spring Boot), and data persistence layer (PostgreSQL/JPA).

## Architecture Overview

```
┌─────────────────────────────────────────────────────────┐
│                    Frontend Layer                       │
│                  Vue.js 2.6 + Axios                     │
│                    (Port 8081)                          │
└─────────────────┬───────────────────────────────────────┘
                  │ HTTP/REST
                  │ JSON
┌─────────────────▼───────────────────────────────────────┐
│                   Backend Layer                         │
│              Spring Boot 2.4.3 REST API                 │
│                  (Port 8080)                            │
│  ┌──────────────┬──────────────┬────────────────┐      │
│  │ Controllers  │   Services   │   Repositories │      │
│  └──────────────┴──────────────┴────────────────┘      │
└─────────────────┬───────────────────────────────────────┘
                  │ JDBC/JPA
                  │ SQL
┌─────────────────▼───────────────────────────────────────┐
│                 Data Persistence Layer                  │
│          PostgreSQL (Production)                        │
│             H2 Database (Testing)                       │
└─────────────────────────────────────────────────────────┘
```

<Note>
  The application follows REST architectural principles with clear separation of concerns between layers.
</Note>

## Data Models

The application uses JPA entities to represent domain objects.

### Part Entity

Core entity representing automotive parts:

```java
@Entity
@Table(name = "part")
public class Part {
    @Id
    @GeneratedValue(strategy = GenerationType.AUTO)
    private Long id;
    
    @Column(name = "name")
    private String name;
    
    @Column(name = "applicationVehicle")
    private String applicationVehicle;
    
    @Column(name = "netWeight")
    private Double netWeight;
    
    @Column(name = "grossWeight")
    private Double grossWeight;
    
    @ManyToMany
    @JoinTable(
        name = "part_vehicle",
        joinColumns = @JoinColumn(name = "part_id"),
        inverseJoinColumns = @JoinColumn(name = "vehicle_id")
    )
    private Set<Vehicle> vehicles = new HashSet<>();
}
```

**Location**: `backend/src/main/java/br/com/tarcnux/spring/jpa/postgresql/model/Part.java:16-38`

**Key Features**:
- **Auto-generated ID**: Uses database sequence for primary key
- **Column Mapping**: Explicit column name definitions
- **Many-to-Many Relationship**: Join table strategy for vehicle associations
- **Bidirectional Relationship**: Helper methods maintain consistency

### Vehicle Entity

Represents vehicles that parts can be associated with:

```java
@Entity
@Table(name = "vehicle")
public class Vehicle {
    @Id
    @GeneratedValue(strategy = GenerationType.AUTO)
    private Long id;
    
    @Column(name = "name")
    private String name;
    
    @ManyToMany(mappedBy = "vehicles")
    @JsonIgnore
    private Set<Part> parts = new HashSet<>();
}
```

**Location**: `backend/src/main/java/br/com/tarcnux/spring/jpa/postgresql/model/Vehicle.java:16-29`

**Key Features**:
- **Inverse Side**: Uses `mappedBy` to indicate Part owns the relationship
- **JSON Serialization**: `@JsonIgnore` prevents circular references
- **Simplified Structure**: Minimal attributes for flexibility

### Entity Relationship Diagram

```
┌─────────────────────┐              ┌─────────────────────┐
│       Part          │              │      Vehicle        │
├─────────────────────┤              ├─────────────────────┤
│ id (PK)             │              │ id (PK)             │
│ name                │              │ name                │
│ applicationVehicle  │              │                     │
│ netWeight           │              │                     │
│ grossWeight         │              │                     │
└──────────┬──────────┘              └──────────┬──────────┘
           │                                    │
           │         ┌──────────────────┐       │
           └────────►│  part_vehicle    │◄──────┘
                     ├──────────────────┤
                     │ part_id (FK)     │
                     │ vehicle_id (FK)  │
                     └──────────────────┘
```

### Relationship Management

Bidirectional relationship helpers ensure data consistency:

```java
public void addVehicle(Vehicle vehicle) {
    this.vehicles.add(vehicle);
    vehicle.getParts().add(this);
}

public void removeVehicle(Vehicle vehicle) {
    this.vehicles.remove(vehicle);
    vehicle.getParts().remove(this);
}
```

**Location**: `backend/src/main/java/br/com/tarcnux/spring/jpa/postgresql/model/Part.java:94-102`

<Warning>
  Always use the helper methods `addVehicle()` and `removeVehicle()` to maintain bidirectional relationship integrity. Direct Set manipulation may cause inconsistencies.
</Warning>

### Service Layer

Data services encapsulate API communication:

```javascript
import http from "../http-common";

class PartDataService {
    getAll() {
      return http.get("/parts");
    }
  
    get(id) {
      return http.get(`/parts/${id}`);
    }
  
    create(data) {
      return http.post("/parts", data);
    }
  
    update(id, data) {
      return http.put(`/parts/${id}`, data);
    }
  
    delete(id) {
      return http.delete(`/parts/${id}`);
    }
  
    deleteAll() {
      return http.delete(`/parts`);
    }
  
    findByName(name) {
      return http.get(`/parts?name=${name}`);
    }

    findByVehicle(vehicle) {
      return http.get(`/parts?vehicle=${vehicle}`);
    }
}

export default new PartDataService();
```

**Location**: `frontend/src/services/PartDataService.js`

**Design Pattern**: Service Layer Pattern
- Centralizes API calls
- Provides reusable methods
- Simplifies component logic
- Uses Axios for HTTP communication

## API Design

The REST API follows RESTful principles and conventions.

### API Versioning

All endpoints are versioned under `/api/v1`:

```java
@RequestMapping("/api/v1")
public class PartController {
    // Endpoints
}
```

This allows for future API evolution without breaking existing clients.

### HTTP Method Mapping

| Operation | HTTP Method | Endpoint Pattern | Example |
|-----------|-------------|------------------|----------|
| List All | GET | `/resource` | `GET /api/v1/parts` |
| Get One | GET | `/resource/:id` | `GET /api/v1/parts/1` |
| Create | POST | `/resource` | `POST /api/v1/parts` |
| Update | PUT | `/resource/:id` | `PUT /api/v1/parts/1` |
| Delete One | DELETE | `/resource/:id` | `DELETE /api/v1/parts/1` |
| Delete All | DELETE | `/resource` | `DELETE /api/v1/parts` |
| Search | GET | `/resource?param=value` | `GET /api/v1/parts?name=polia` |

### Response Status Codes

The API uses appropriate HTTP status codes:

- **200 OK**: Successful GET, PUT operations
- **201 CREATED**: Successful POST operation
- **204 NO CONTENT**: Successful DELETE or empty result set
- **404 NOT FOUND**: Resource not found
- **500 INTERNAL SERVER ERROR**: Server-side error

**Implementation Example**:

```java
@GetMapping("/parts")
@ResponseStatus(HttpStatus.OK)
public ResponseEntity<List<Part>> getAllParts(...) {
    if (parts.isEmpty()) {
        return new ResponseEntity<>(HttpStatus.NO_CONTENT);
    }
    return new ResponseEntity<List<Part>>(parts, HttpStatus.OK);
}
```

**Location**: `backend/src/main/java/br/com/tarcnux/spring/jpa/postgresql/controller/PartController.java:43-71`

## Security Considerations

### CORS Configuration

Cross-Origin Resource Sharing is explicitly configured:

```java
@CrossOrigin(origins = "http://localhost:8081")
```

This restricts API access to the frontend application running on port 8081.

### Database Configuration

Database credentials should be configured in `application.properties`:

```properties
spring.datasource.url=jdbc:postgresql://localhost:5432/partsCatalog
spring.datasource.username=postgres
spring.datasource.password=postgres
```

<Warning>
  In production environments, use environment variables or secure configuration management for database credentials instead of hardcoding them.
</Warning>
